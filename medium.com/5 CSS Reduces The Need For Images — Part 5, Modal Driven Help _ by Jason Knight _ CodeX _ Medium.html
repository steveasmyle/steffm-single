Then there’s the templates object used to hold… well, template functions. The first function:

    template = {
    
      keyCell : (data, tagName) => {
        const
          td = document.createElement("td");
          values = data[tagName] instanceof Array ? data[tagName] : [data[tagName]],
          className = data[tagName + "Class"] || "";
        for (value of values) {
          if (value) td.__make(tagName, { className }, [ "span", value ]);
          else td.__make("em", "none");
          className = null;
        }
        return td;
      }, // template.keyCell

…creates a table cell containing what looks like a key,. It is set up to either use a string if present, or to make a bunch of them out of an array of strings. The tag placed inside the TD can be set at call-time since the “key” column will use <kbd> but the on-screen key uses <samp>

      keyCellTables : (docData) => {
        let result = [];
        for (const [className, data] of Object.entries(docData)) {
          const tbody = document.__make("tbody");
          for (const button of data.buttons) {
            tbody.__make(
              "tr", 
              template.keyCell(button, "kbd"),
              template.keyCell(button, "samp"),
              [ "th", { scope : "row" }, button.desc ]
            );
          }
          result.push(document.__make(
            "table",
            { className },
            ["caption", data.caption ],
            tbody
          ));
        }
        return result;
      }, // template.keyCellTables

Generates the entire table within a documentation section. First it makes the TBODY and populates its rows, then it appends that TBODY to a generated TABLE with the CAPTION assigned.

      flag : (docData) => { 
        let result = [];
        for (const [className, data] of Object.entries(docData)) {
          const
            table = document.__make("table"),
            article = document.__make(
              "article",
              { className },
              [ "h3", data.title ]
            );
          for (const [flag, desc] of data.flags) table.__make(
            "tr",
            [ "th", { scope : "row" }, flag ],
            [ "td", desc ]
          );
          if (data.desc) article.__make("p", data.desc);
          article.appendChild(table);
          result.push(article);
        }
        return result;
      }, // template.flag

The flag template isn’t too complex either. It just goes through and makes an article containing an H3 for the title, and a table containing the rows of the flag and its description. Again whilst this is only two columns this IS tabular data, treat it as such. The “description” field is optional, and if present is inserted as a paragraph between the H3 and the table.

Why an H3? Because the modal is at H2 depth and this is a subsection of that modal. That’s what H3 MEANS, the start of a subsection of the H2 preceeding it. <broken record>Which as I’m always saying is why skipping over heading depths going down or pairing H1+H2 or H3+H4 for “heading and tagline” is utterly, totally, and unequivically ignorant of what these tags are even for!</broken>

      indexLink : (ul, hash, className, ...contents ) => {
        const
          a = document.__make("a", { href : "#" + hash }, ...contents),
          li = ul.__make("li", a);
        if (className) li.className = className;
        return a;
      }, // template.indexLink

Makes an LI > A, optional classname on the LI, extra children are contents of the anchor. Returns the anchor for other things like event hooks if needed.

      index : (parent, id) => {
        const ul = document.createElement("ul");
        for (const [indexId, indexData] of Object.entries(docModals)) {
          if (id !== indexId) template.indexLink(ul, indexId, null, indexData.construct[0]);
        }
        parent.__make("nav", { className : "helpIndex" }, ul);
      }, // template.index

Creates the primary index of all help pages using the docModals object. Omits the passed ID from the list, which is typically the page this index is shown on.

      pagination : (parent, id) => {

          ul = document.createElement("ul");

          ul, linkIds.prev, "prev", docModals[linkIds.prev].construct[0]
        );
        if (id !== docModalIndex) template.indexLink(
          ul, docModalIndex, "index", docModals[docModalIndex].construct[0]
        );
        if (linkIds.next) template.indexLink(
          ul, linkIds.next, "next", docModals[linkIds.next].construct[0]

          "footer",
          [ "nav", { className : "helpPagination" },  ul ]
        );

The CSS

I’m not going to go nuts documenting this. If you really want to understand how I do modal, please read my earlier article on the topic.

https://levelup.gitconnected.com/modal-dialog-driven-websites-without-javascript-16e858615780

These modal dialogs and navigation are powered by :target and hash links. The style for just the modals — assuming a reset is in use — goes something like this:

#fauxBody,
.modal,
.modal > a {
  position:fixed;
  bottom:0;
  right:0;
  height:100%;
  width:100%;
  overflow:auto;
}

.modal > a span,
. modal > div > h2 a span {
  position:absolute;
  left:-200vw;
}

#fauxBody is needed around our content to prevent the “double scrollbar” bug common when the window is shorter than the modal and content’s height. If we just position:fixed and left body alone, the body scrollbar is subtracted from the fixed element’s size. All three of these initial elements share the same sizing information, so group them together.

The span inside the modal are mostly there to make “Google Lighthouse” sierra tango foxtrot uniform about “accessibility”. In reality a user on an accessibility UA would never get this far in the first place, but it never hurts to have actual text inside an anchor. Those anchors exist to close the menu, either by clicking outside the centered inner DIV (.modal > a) or by clicking on a giant X (.modal > div > h2 > a)

.modal {
 z-index:999;
 right:100vw;
 display:flex;
 align-items:center;
 justify-content:center;
 padding:1rem;
 opacity:0;
 transition:right 0s 0.5s, opacity 0.5s;
 background:
  #2484
  radial-gradient(
   at center,
   #FFF4,
   #0484
  );
}

.modal:target {
 right:0;
 opacity:1;
 transition:opacity 0.5s;
}

We position this off the screen so we can use delayed animations on the show/hide, allowing the opacity fade to show.

.modal:target > a,
.modal h2 a {
  display:block; /* undo hidden state */
  text-decoration:none;
}

When it’s visible, we want to undo the hidden state. That hidden is set because for non-visual users like braille readers, these are just more sections of the page.

.modal > div {
  margin:auto;
  overflow:hidden;
  background:#DEF;
  border:0.0625rem solid #0008;
  border-radius:0.5em;
  box-shadow:
    0 0 2rem #0004,
    0.125rem 0.125rem 0.5rem #0004;
  scale:0.1;
  transition:scale 0.5s;
}

.modal:target > div {
  scale:1;
}

The inner DIV is nothing fancy, though it gets a scaling animation when it comes into or leaves view.

.modal h2 {
  padding:0.25rem 1rem;
  background:#CDE;
  border-bottom:0.0625rem solid #0008;
}

.modal h2 a {
  float:right;
  margin-left:1rem;
}

.modal h2 a:after {
  content:"\F00D";
  font-family:interface;
  color:#000;
}

.modal p {
  margin:1rem;
}

The heading and its close anchor, easy-peasy lemon-squeezy.

From there it’s all just style inside the modals which is not significantly different from styling any other subsections of a website. I’m not going to go any deeper explaining it as the help system was not part of the initial point or focus of this series of articles.

Conclusion

It’s amazing how CSS can be used as a drawing tool. It makes sense though as layered elements of different sizes, shapes, and colouration is what most of the world is perceived as. Shading, shadows, these are all possible. Likewise as CSS gains more and more capability, JavaScript becomes inscreasingly less needed. Look at the modal dialog help system where you could write them without a lick of JavaScript involved… or at most that little bit of scripting to close them if you hit “esc” while they’re open.

There are so many things we used to waste large raster images, or play with SVG, or throw large amounts of scripting at which just don’t need JS for anymore, reducing the overhead, cost, and difficulty of both creating and maintianing your sites and apps.

Freeing up room for the JavaScript you do need to better do its job.
